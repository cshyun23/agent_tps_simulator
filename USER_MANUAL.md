## 시작하기

### 애플리케이션 접속

1. 브라우저를 열고 **`http://localhost:5173`** 주소창에 입력
2. Agent TPS Simulator 홈페이지가 로드됨
3. 상단 탭으로 3가지 화면에 접근 가능:
   - 🔷 **Agent Flow** - 플로우 설계 및 시뮬레이션
   - 🔶 **LLM Hub** - 가상 LLM 서버 관리
   - 🔸 **비교 뷰** - 시뮬레이션 결과 비교

### 첫 번째 사용 팁

처음 사용하는 경우, 이 순서로 진행하세요:

1. **LLM Hub 탭에서 가상 LLM 서버 생성** (5분)
2. **Agent Flow 탭에서 샘플 플로우 선택** (1분)
3. **시뮬레이션 실행** (3분)
4. **결과 확인** (2분)

## 기본 개념

### Agent Flow란?

Agent Flow는 **AI 에이전트가 처리해야 할 작업 흐름**을 나타냅니다.

예시:
```
사용자 질문 → 질문 분석 (LLM) → 웹 검색 (Tool) → 답변 생성 (LLM) → 결과 반환
  START          LLM            TOOL           LLM           END
```

### 노드 (Node)란?

플로우를 구성하는 **작업 단위**입니다:

| 노드 | 설명 | 처리 시간 |
|------|------|----------|
| 🟢 **Start** | 플로우 시작점 (사용자 요청 진입) | - |
| 🔵 **LLM** | LLM 모델로 추론 (예: Claude, GPT) | TTFT + 토큰 생성 시간 |
| 🟡 **Tool** | 외부 도구 호출 (웹 검색, API 등) | 사용자 설정 시간 |
| 🟠 **Conditional** | 조건에 따라 다른 경로로 분기 | 즉시 (분기 확률 적용) |
| 🟣 **Parallel** | 여러 노드를 동시 실행 | 가장 오래 걸리는 노드 시간 |
| 🔴 **End** | 플로우 종료점 | - |

### TPS와 Latency

- **TPS (Transactions Per Second)**: 초당 처리 요청 수
  - 높을수록 좋음 (예: 50 TPS = 1초에 50개 요청 처리)
- **Latency (지연시간)**: 요청 처리에 걸리는 시간
  - 낮을수록 좋음 (예: 250ms = 0.25초)

## 플로우 만들기

### 화면 레이아웃

```
┌─────────────────────────────────────────────────────┐
│  [Agent Flow]  [LLM Hub]  [비교 뷰]                 │ ← 탭 전환
├──────────┬─────────────────────────┬────────────────┤
│ 플로우   │   플로우 에디터         │  시뮬레이션    │
│ 목록     │   (노드/엣지 편집)      │  패널          │
│ (40%)    │       (40%)             │   (20%)        │
└──────────┴─────────────────────────┴────────────────┘
```

### 1단계: 플로우 선택 또는 생성

**왼쪽 사이드바에서:**

1. 기존 플로우를 사용하려면: 플로우 이름 **클릭**
2. 새 플로우를 만들려면: **`+ 새 플로우`** 버튼 클릭

#### 새 플로우 생성 시 템플릿 선택

템플릿을 선택하면 기본 구조가 자동으로 만들어집니다:

**빈 플로우**
```
[START] → [END]
```
- 처음부터 직접 설계하고 싶을 때 선택

**단순 RAG**
```
[START] → [LLM: 질문 분석] → [TOOL: 검색] → [LLM: 답변] → [END]
```
- 검색 기반 질문 답변 시스템

**ReAct 루프**
```
[START] → [LLM] → [Conditional] ⇄ [TOOL] → [END]
         (루프백)
```
- 반복적 문제 해결 (도구 사용)

**병렬 Tool 호출**
```
[START] → [PARALLEL] → [TOOL1] ┐
                    ├→ [TOOL2] ├→ [GATHER] → [END]
                    └→ [TOOL3] ┘
```
- 여러 도구를 동시에 호출

### 2단계: 노드 추가

에디터 상단의 버튼으로 노드 추가:

```
+ Start  |  + LLM  |  + Tool  |  + Cond  |  + Para  |  + End
```

예: RAG 시스템 만들기
1. 이미 START와 END가 있음
2. **`+ LLM`** 클릭 → 에디터 중앙에 새 LLM 노드 생성
3. **`+ TOOL`** 클릭 → 또 다른 노드 생성
4. 필요하면 반복

### 3단계: 노드 연결

**노드 연결 방법:**

1. 첫 번째 노드의 **오른쪽 동그란 점**(핸들) 클릭 & 드래그
2. 두 번째 노드의 **왼쪽 점**으로 드래그 해서 놓기
3. 엣지(화살표)가 생성됨

**루프 만들기** (같은 노드로 돌아가기):
- Conditional 노드의 한 분기를 이전 노드로 연결하면 점선 표시

### 4단계: 노드 설정

**노드 클릭 → 오른쪽에 설정 팝업 표시**

#### LLM 노드 설정

| 설정 | 설명 | 예시 |
|------|------|------|
| **LLM 서버** | 어떤 LLM을 사용할지 선택 | "Claude Opus" |
| **입력 토큰** | 평균 입력 토큰 수 | 512 |
| **출력 토큰** | 평균 출력 토큰 수 | 128 |
| **폴백 지연** | 서버 미선택 시 처리 시간(ms) | 1000 |
| **최대 동시 수** | 동시에 처리할 요청 수 | 4 |

💡 **팁**: 입력/출력 토큰은 실제 서비스의 평균값을 입력하세요

#### Tool 노드 설정

| 설정 | 설명 | 예시 |
|------|------|------|
| **처리 시간** | 도구 실행에 걸리는 시간(ms) | 500 |

#### Conditional 노드 설정

이 노드에서 요청을 분기합니다. 예: "질문 유형에 따라 다른 처리"

**분기 추가하기:**

1. **`+ 추가`** 버튼 클릭
2. 각 분기별로 입력:
   - **분기 이름**: "의도 명확함"
   - **확률(%)**: "60" (60%의 요청이 이 분기로 감)
   - **대상 노드**: 다음 노드 ID (예: "node_5")

⚠️ **중요**: 모든 분기의 확률 합이 정확히 **100%**여야 합니다!

### 5단계: 플로우 저장

상단 중앙의 **`💾 Save`** 버튼 클릭

✅ 토스트 메시지 "플로우가 저장되었습니다"가 표시되면 완료

### 플로우 관리

**사이드바 플로우 항목:**

| 작업 | 방법 |
|------|------|
| 선택 | **클릭** |
| 이름 변경 | **더블클릭** → 이름 입력 → Enter |
| 복제 | Hover → **`📋 복제`** 버튼 |
| 삭제 | Hover → **`🗑 삭제`** 버튼 |

**가져오기/내보내기:**

상단 중앙:
- **`📤 Export`**: 플로우를 JSON 파일로 저장 (다른 컴퓨터에서 사용 가능)
- **`📥 Import`**: 저장된 JSON 파일에서 플로우 복원

## LLM 서버 설정

### 왜 서버를 등록해야 할까?

시뮬레이션에서 LLM의 처리 시간을 정확하게 계산하려면:
- **TTFT** (Time To First Token): 첫 토큰이 나올 때까지 걸리는 시간
- **TPOP** (Time Per Output Token): 토큰 하나당 생성 시간

이 정보를 알아야 정확한 시뮬레이션이 가능합니다.

### 서버 생성하기

**LLM Hub 탭 → `+ 서버 추가` 버튼**

#### 정보 입력

| 필드 | 설명 | 입력 예 |
|------|------|--------|
| **이름** | 서버 식별명 (UI에 표시) | "Claude Opus" |
| **모델** | 모델 이름 | "Claude-3.5-Sonnet" |
| **파라미터** | 모델 크기 (B = 10억) | 70 |
| **GPU 종류** | 사용하는 GPU 타입 | "H100-80GB" |
| **GPU 개수** | 보유한 GPU 수 | 4 |
| **Ref TTFT** | 참고 TTFT (512 토큰 기준, ms) | 85 |
| **Ref TPOP** | 참고 TPOP (ms/토큰) | 18 |

#### 값 구하는 방법

**실제 서비스에서 측정하는 경우:**

1. 프로덕션 서버 로그 확인
2. 평균 처리 시간 계산
3. 입력 토큰 수와 생성 토큰 수 측정

**벤치마크 데이터 사용:**
- 공개된 LLM 벤치마크 참고
- 제조사 공식 문서 확인

**예시: Claude Opus (H100 4대)**
```
TTFT: 85ms (512 토큰 입력 기준)
TPOP: 18ms/토큰
```

### 서버 관리

각 서버 카드의 버튼:
- **`✏ 편집`**: 서버 정보 수정
- **`🗑 삭제`**: 서버 삭제

### GPU 레퍼런스

하단 "GPU 레퍼런스" 섹션:
- **`📤 Export`**: GPU 데이터 다운로드
- **`📥 Import`**: 저장된 데이터 업로드

## 시뮬레이션 실행

### 시뮬레이션이란?

플로우가 **실제로 요청을 처리한다면 어떻게 될까**를 시뮬레이션합니다.

예를 들어:
- 동시에 100명의 사용자가 요청을 보냈을 때
- 각 노드의 큐(대기열)는 어떻게 될까?
- 처리 시간은?
- 실패율은?

### 플로우 선택

**Agent Flow 탭:**

1. 왼쪽 사이드바에서 플로우 선택
2. 플로우 에디터에 로드됨

### 시뮬레이션 설정

**오른쪽 패널 "Simulation Panel"에서:**

#### 기본 설정

**1️⃣ 도착 패턴 (사용자가 들어오는 패턴)**

**Ramp-up 선택:**
```
사용자 수 ▲
    100 |     ─────────  (Hold Duration)
        |    /
        |   / (Ramp Up)
    0   |__/─────────────► 시간
        0    10s    40s
     (Ramp Duration)
```

입력:
- **Peak Users**: 최대 동시 사용자 (예: 100)
- **Ramp Duration**: 증가 시간(초) (예: 10)
- **Hold Duration**: 유지 시간(초) (예: 30)
- **Ramp Shape**: Linear (일직선) 또는 Smooth (곡선)

**Wave 선택:**
```
사용자 수 ▲
    100 |  ╱╲    ╱╲    ╱╲
        | ╱  ╲  ╱  ╲  ╱  ╲
    10  |╱────╲╱────╲╱────╲
        └─────────────────────► 시간
```

입력:
- **Min Users**: 최소 사용자 (예: 10)
- **Peak Users**: 최대 사용자 (예: 100)
- **Period**: 주기(초) (예: 20)

**2️⃣ 총 시간**

시뮬레이션이 진행될 시간(초)

예: 60초 = 1분 시뮬레이션

**3️⃣ 재생 속도**

실제보다 몇 배 빠르게 시뮬레이션할지

- **1x**: 실시간 (1초 = 1초 시뮬레이션)
- **2x**: 2배 빠름 (1초 = 2초 시뮬레이션)
- **5x**: 빠른 테스트용
- **10x**: 매우 빠른 테스트

💡 **팁**: 초기 테스트는 **10x**로 빠르게 확인하세요

**4️⃣ 메트릭 표시**

- ☑️ **P95 메트릭 표시**: 95% 사용자의 대기 시간
- ☑️ **P99 메트릭 표시**: 99% 사용자의 대기 시간

⚠️ **중요**: P99는 최악의 경우를 보여줍니다. 이것이 서비스 품질의 핵심!

### 시뮬레이션 실행

1. 플로우 에디터에서 **Start 노드** 클릭
2. **End 노드(들)** 선택 (여러 개 가능)
3. 상단 우측 **`▶ Run Simulation`** 버튼 클릭
4. 설정 후 버튼 클릭으로 시작

### 실시간 모니터링

시뮬레이션 진행 중:

```
⏳ 시뮬레이션 진행 중...
재생속도: 10x
```

**실시간 차트 표시:**
- 파란선: **TPS** (초당 처리 요청 수)
- 초록선: **E2E Latency** (평균 지연시간)
- 주황선: **P95** (95% 지연시간)
- 빨강선: **P99** (99% 지연시간)

**제어:**
- **`⏸ 일시정지`**: 시뮬레이션 멈춤
- **`⏹ 초기화`**: 시뮬레이션 중단 및 초기화

### 결과 요약

시뮬레이션 완료 후:

```
📊 요약
─────────────────────────
처리됨: 3000개
실패: 5개 (0.2%)

E2E Latency: 248ms
Peak TPS: 52.1

P95: 352ms
P99: 451ms
```

**각 항목의 의미:**

| 항목 | 의미 | 목표 |
|------|------|------|
| **처리됨** | 성공한 요청 수 | 높을수록 좋음 |
| **실패** | 실패한 요청 수 | 낮을수록 좋음 |
| **E2E Latency** | 평균 처리 시간 | 낮을수록 좋음 |
| **Peak TPS** | 최고 처리량 | 높을수록 좋음 |
| **P95/P99** | 느린 요청의 시간 | 낮을수록 좋음 |

### 결과 다운로드

**`📥 JSON`** 또는 **`📥 CSV`** 버튼으로:
- **JSON**: 전체 상세 데이터 (프로그래밍 분석용)
- **CSV**: 시간대별 메트릭 (Excel/Sheets 분석용)

## 결과 분석

### 여러 구성 비교하기

**시나리오:**
- 구성 A: Peak 100명, Ramp 10초
- 구성 B: Peak 150명, Ramp 5초

→ 어느 것이 더 나은 성능인가?

### 비교 뷰 사용

**비교 뷰 탭에서:**

#### 1단계: 결과 선택

왼쪽 "시뮬레이션 히스토리"에서:
- 비교할 결과의 **체크박스** 클릭 (2개 이상 선택)
- 각 결과의 날짜 확인 가능

#### 2단계: 비교 실행

**`비교 (선택 수)`** 버튼 클릭

#### 3단계: 차트 분석

4가지 차트로 비교:

**1) 요청 처리**
```
완료 수 ▲
   3500│  ├─ A  ├─ B
   3000│  │     │  
        │  ├─ 실패
```
→ A가 더 많은 요청 처리

**2) TPS 비교**
```
Peak TPS ▲
      55│ A ┐
      50│   │ B
      45│   │
```
→ A가 더 높은 처리량

**3) Latency 비교**
```
시간(ms) ▲
      300│ A ║  B
      250│   ║
      200│   ║ E2E
           P95 상세
```
→ A가 더 빠른 응답

**4) 백분위수**
```
시간(ms) ▲
      500│   P99
      400│ ┌─┐ A
      300│ │ │ B
           P95
```
→ A의 최악의 경우가 더 나음

#### 4단계: 요약 테이블

| 항목 | A | B | 차이 |
|------|---|---|------|
| Completed | 3000 | 2800 | A가 7% 좋음 |
| Failed | 5 | 15 | A가 3배 나음 |
| E2E (ms) | 248 | 320 | A가 23% 빠름 |
| Peak TPS | 52.1 | 48.5 | A가 7% 높음 |
| P95 (ms) | 352 | 425 | A가 17% 좋음 |
| P99 (ms) | 451 | 580 | A가 22% 좋음 |

**결론**: 구성 A가 모든 지표에서 더 우수함

### 최적화 전략

**성능이 좋지 않을 때:**

1. **TPS가 낮으면**:
   - LLM의 최대 동시 처리 수 증가
   - LLM 서버의 GPU 수 증가
   - TTFT/TPOP 개선

2. **Latency가 높으면**:
   - Tool 노드의 처리 시간 감소
   - 병렬 처리 추가 (Parallel 노드)
   - 불필요한 단계 제거

3. **실패율이 높으면**:
   - Conditional 노드의 재시도 설정
   - 폴백 지연 시간 증가
   - 서버 안정성 개선

## 자주 묻는 질문

### Q1: 토큰이란 무엇인가요?

**A:** LLM이 처리하는 텍스트의 최소 단위입니다.

예시:
```
"Hello, world!" = 약 3-4개 토큰
"안녕하세요" = 약 2-3개 토큰 (한국어)
```

**입력/출력 토큰 수 결정:**
1. 실제 서비스 로그 분석
2. 평균값 입력 (예: 평균 512개 입력)
3. 표준편차 자동 계산 (편차 = 평균 × 0.2)

### Q2: TTFT와 TPOP의 차이는?

**A:** 

**TTFT** (Time To First Token): 
- 처음 답변이 나올 때까지 걸리는 시간
- 사용자가 기다리는 체감 지연시간
- 입력 토큰이 많을수록 증가

**TPOP** (Time Per Output Token):
- 토큰 하나를 생성하는데 걸리는 시간
- 답변 길이가 길수록 증가

### Q3: Concurrent(동시 처리 수)는 몇으로 설정해야 하나요?

**A:** GPU의 VRAM과 KV Cache 크기에 따릅니다.

**일반적인 값:**
- 소형 모델 (7B): 8-16
- 중형 모델 (30B): 4-8  
- 대형 모델 (70B): 2-4

**확인 방법:**
1. 실제 서버에서 "최대 동시 요청"을 테스트
2. 메모리 부족 에러 직전의 수치 사용
3. 보수적으로 설정 (예: 최대의 80%)

### Q4: 플로우 저장이 안 되는데요?

**A:** 다음을 확인하세요:

1. **백엔드 서버 실행 여부**: `http://localhost:8000/health` 접속
2. **브라우저 콘솔 확인**: F12 → Console 탭 → 에러 메시지
3. **플로우 유효성**:
   - Start 노드와 End 노드 모두 있는가?
   - 모든 노드가 연결되었는가?
   - Conditional 분기 확률의 합이 100%인가?

### Q5: 시뮬레이션 결과가 비정상적이에요

**A:** 다음을 확인하세요:

| 증상 | 원인 | 해결 |
|------|------|------|
| TPS가 0에 가까움 | LLM 설정 오류 | TTFT/TPOP 값 확인 |
| Latency가 너무 높음 | 토큰 수 설정 오류 | 입력/출력 토큰 감소 |
| 실패율 100% | 플로우 설계 오류 | Start→End 경로 확인 |
| 데이터가 없음 | 시뮬레이션 미실행 | 다시 실행 |

### Q6: 여러 번 테스트하고 싶어요

**A:** 이렇게 하세요:

1. **플로우 복제**: 사이드바 Hover → `📋 복제`
2. **원본은 보존**: 다양한 버전 유지 가능
3. **각각 시뮬레이션 실행**: 결과 자동 저장
4. **비교 뷰에서 비교**: 최고 성능 구성 찾기

### Q7: 결과를 Excel로 분석하고 싶어요

**A:** 

1. 시뮬레이션 완료 후 **`📥 CSV`** 다운로드
2. Excel 또는 Google Sheets에서 열기
3. 차트 생성 가능

**CSV에 포함된 데이터:**
- sim_time_sec: 시뮬레이션 시간
- overall_tps: 당시의 TPS
- e2e_avg_latency_ms: 평균 지연시간
- e2e_p95_latency_ms: P95
- e2e_p99_latency_ms: P99
- total_completed: 누적 완료 요청
- total_failed: 누적 실패 요청

### Q8: 플로우를 다른 사람과 공유하고 싶어요

**A:**

1. 플로우 **`📤 Export`** 클릭
2. JSON 파일 다운로드됨
3. **파일 메일/채팅으로 전송**
4. 상대방: **`📥 Import`** 클릭 → 파일 선택

### Q9: 성능을 개선하려면 어떻게 하나요?

**A:** 단계별 테스트:

1. **현재 상태 측정**: 기본 구성으로 시뮬레이션
2. **한 가지만 변경**: 예) Ramp duration 5→10초
3. **결과 비교**: 성능 개선되었나?
4. **반복**: 최적값 찾기

**일반적인 최적화:**
- LLM 최대 동시 수 증가
- Tool 노드의 병렬화
- 불필요한 단계 제거
- Conditional의 효율적 분기

### Q10: 에러 메시지가 떴어요

**A:** 브라우저 콘솔 확인 (F12):

**"Cannot connect to server"**
→ 백엔드 실행: `python main.py`

**"Flow not found"**
→ 플로우 다시 로드: 새로고침 (Ctrl+R)

**"Invalid flow configuration"**
→ 플로우 유효성 검증 재실행
