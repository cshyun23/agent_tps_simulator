전체 설계를 꼼꼼히 분석할게요. 카테고리별로 정리합니다.

분석 결과
1. 시뮬레이션 엔진 — 핵심 누락
① 시뮬레이션 방식 미정의 (가장 중요)
현재 설계에는 시뮬레이션 엔진이 어떤 방식으로 시간을 진행시키는지 정의되어 있지 않습니다. 두 가지 방식이 있고 선택이 필요합니다.

방식	설명	특징
DES (Discrete Event Simulation)	이벤트 발생 시점으로만 시간 도약	정확, 빠름, 복잡
Time-step	고정 시간 단위(예: 10ms)로 순차 진행	단순, 오버헤드 있음
이 프로젝트에는 DES 방식이 적합합니다. 큐 모델 + 이벤트 기반이 자연스럽고 재생 속도 제어도 구현하기 좋습니다.

② TTFT 계산에 배치 대기 누락
현재 TTFT 공식은 input token 수에만 선형 비례하는데, vLLM continuous batching에서는 다른 요청의 prefill이 진행 중이면 내 prefill은 그 뒤에 순서를 기다립니다. 따라서 실제로는:


실제 TTFT = prefill 대기 시간 (이전 prefill 완료까지) + 내 prefill 처리 시간
이 부분이 빠지면 고부하 시 TTFT가 실제보다 훨씬 낙관적으로 나옵니다.

③ KV Cache 동적 증가 미반영
현재는 max_concurrent_requests를 고정값으로 계산하는데, 실제로는 각 요청이 decode하면서 KV cache를 점점 더 많이 씁니다. 요청이 절반쯤 생성됐을 때 다른 요청을 받을 수 있는 여유가 줄어드는 효과가 없습니다.

④ 같은 LLM 서버를 여러 노드가 공유할 때
현재 설계는 노드마다 concurrency가 독립적인 것처럼 보이는데, 여러 LLM 노드가 동일한 LLM 서버를 바라볼 경우 KV cache와 처리 용량을 공유해야 합니다. 이 부분 처리 방식이 없습니다.

2. 플로우 설계 — 빠진 노드 타입 및 제약
① Parallel 노드 없음
실제 에이전트에서 가장 흔한 패턴인 **병렬 실행 (fan-out / fan-in)**이 없습니다.


[노드A] ─┬─→ [Tool1] ─┐
         └─→ [Tool2] ─┴─→ [집계 노드]
여러 Tool을 동시에 호출하고 결과를 모아 다음 단계로 가는 구조입니다.

② 루프 무한 반복 방지 미정의
Conditional에서 이전 노드로 루프백이 가능한데, 요청 하나가 최대 몇 번까지 루프를 돌 수 있는지 제한이 없습니다. 시뮬레이션에서 무한 루프가 발생할 수 있습니다.

③ 플로우 유효성 검증 없음
시뮬레이션 시작 전 검사해야 할 항목이 정의되어 있지 않습니다.

분기 확률 합계가 100%인지
Start/End 노드가 각각 하나씩 있는지
연결되지 않은 고립 노드가 없는지
End에 도달하지 않는 데드엔드 경로가 없는지
④ 토큰 수 고정값 문제
input/output 토큰을 단일 정수로만 설정하는데, 실제 트래픽에서는 요청마다 토큰 수가 다릅니다. **분포(평균 ± 표준편차, 또는 min/max 범위)**를 지정하면 훨씬 현실적인 시뮬레이션이 됩니다.

3. 메트릭 — 실무에서 필수인 것들
① 평균만 있고 백분위수 없음
현재 메트릭은 모두 평균(avg)입니다. 성능 분석에서는 P50 / P95 / P99 latency가 핵심입니다. 평균이 좋아도 P99가 10초면 실제 서비스는 문제입니다.

② 시뮬레이션 종료 후 요약 리포트 없음
시뮬레이션이 끝났을 때 전체 결과를 요약해주는 뷰가 없습니다.

전체 처리 요청 수 / 실패 수
노드별 peak 큐 깊이
병목 구간 분석
P50/P95/P99 종합
③ 결과 내보내기 없음
시뮬레이션 결과를 CSV/JSON으로 다운로드하는 기능이 없습니다. 다른 툴에서 분석하거나 기록 보관이 안 됩니다.

④ 시뮬레이션 비교 없음
구성을 바꿔가며 여러 번 시뮬레이션할 텐데, 이전 실행 결과와 비교하는 기능이 없습니다.

4. UI/UX — 누락된 기능
① 플로우 목록 관리 미정의
여러 플로우를 만들 수 있는지, 있다면 목록을 어떻게 보여주는지 정의가 없습니다.

② Undo/Redo 없음
플로우 편집에서 실수 시 되돌리기가 없습니다.

③ 노드 템플릿/프리셋 없음
자주 쓰는 패턴(단순 RAG, ReAct 루프 등)을 빠르게 시작할 수 있는 템플릿이 없습니다. 없어도 되지만 UX를 크게 개선합니다.

5. 백엔드 / 운영 — 고려 필요
① WebSocket 프로토콜 미정의
실시간 메트릭을 어떤 형태로, 얼마나 자주 보내는지 정의가 없습니다. 메시지 포맷이 필요합니다.

② 시뮬레이션 세션 관리 없음
